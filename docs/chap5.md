# 型を使ったドメインモデリング
- ドメインエキスパートがソースコード内でのドメインモデルの定義をメンタルモデルとして共有できるのが理想
    - メンタルモデル -> ソースコード間の変換が不要なので
- ドメインモデルのパターンを観察して典型的なパターンを見つける
    - 単純な値
        - 文字列,整数といったプリミティブな値で考えるのではなく、ユビキタス言語で考えるべき
            - 全ての文字列型の値が有効ではなく、ドメイン内での制約によりそのうちの一部のみが有効なため
    - 値の組み合わせ
        - ひとまとまりで扱われる値の組み合わせ
    - 選択肢
    - ワークフロー
- 典型的なパターンごとのモデリング
    - 単純な値
        - ドメインモデルごとにプリミティブな型のラッパーを作る
            - F#においては選択肢が一つしかない共用型を定義する
        - これにより異なるドメインモデルを参照する処理や型の定義を明示的に分けられる
            - これを行わない場合プリミティブな型をそのまま指定する->意図しない互換性が生まれてしまう
        - メモ:単一ケース共用体の分解 = 他言語での値の代入という理解で正しい？
    - 値の組み合わせ
        - AND型
    - 選択肢
        - OR型
    - ワークフロー
        - 関数で表現
        - 複数の値の引数として値間の関係によって定義の方法を変えるとよい
            - 両値とも入力の場合
                - AND型, または複数引数の定義
            - 依存している場合（処理に必要だが入力ではない場合）
                - 複数引数の定義
        - エフェクト（関数が主な出力以外に行うこと）の表現  
            - Result型, Async型などで表現
                - 成功と失敗を型で表現
- 値オブジェクト、エンティティの表現
    - 値オブジェクト
        - 等価性が値の一致のみで判断されるもの
    - エンティティ
        - 等価性が値の一致以外のものでも判断されるもの
    - これらは業務内でのタイミングに依存する
    - F#での表現
        - 値オブジェクト
            - そのまま
        - エンティティ
            - 識別子をAND型に持たせ、値の等価性の判定関数を変更する
            - 値の変更時は識別子が同一なコピーを作成->値を変更する
- 集約
    - エンティティには階層構造がある
    - 下位のエンティティの変更を行う場合、上位のエンティティを介して変更する
        - 値が不変な言語で実装しているとこの方針が強制される
        - 整合性, 不変条件を維持する境界として機能する
        - このルールにより集約のルートを意識して分割して設計できる
            - 不用意に下位に含めてしまうと単一のルートのエンティティに全てが集約される
        - 永続化、送受信（データベーストランザクション）の単位でもある
