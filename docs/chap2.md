# ドメインの理解
ある特定のビジネスワークフローを取り上げそれを理解する方法の解説
例：注文のビジネスワークフローを例に考える

- ドメインエキスパートのインタビューから理解していく
    - 先入観をもたず観察から理解していく
        - 対象顧客に対する最適化を見逃さないため
- ビジネスワークフローのインプットとアウトプット
    - インプット
        - 何でも良い
        - 例：注文書など
    - アウトプット
        - 別のコンテキストに送られるイベント
        - 例：注文が完了したイベント
    - 副作用
        - アウトプットに含まれないもの以外の生成物
- 技術的な先入観を持たず、あるがままのドメインの理解に徹する
    - 永続性非依存の原則を守る
        - DDDの用語
        - 永続化をどう行うかは考慮せずモデリングを行う（RDB形式にとらわれない）
    - クラス駆動設計を重視しない
        - 技術依存の一種
        - ドメインエキスパートの語彙の中に存在しないものを存在させない
            - 基底クラスなど
- 複雑さをドメインモデルで表現する技術
    - 文書でワークフローを表現する
        - ある程度形式的な文書とし、ドメインエキスパートに見てもらう
    - 制約条件の表現
        - 解決空間に現れる語彙のうち、値に関しての制約を表現する
        - プリミティヴなStringや数値型に制約を加える形で表現する
            - 不明な点があればつ度ドメインエキスパートに確認する
    - 注文のライフサイクルを表現
        - ワークフローの中で表現される状態について別々の名前、定義を与えておく
            - 「ドメインエキスパートの中に存在しない語彙を使わない」制約を破っている気がするが、あくまでも状態への参照なのでOK？（表している状態については明示的にドメインエキスパートが言及しているので）
        - ワークフローのステップの明確化で入出力を明確にできる利点がある
    - ワークフローのステップの具体化
        - 各ステップを形式的な文章で記載する
        - 入力、出力、内容を明示的に書く
