# DDDの紹介
ここではDDDの定義、解決したい問題などを述べる。

## 前提
- ソフトウェアのユーザーに実際に提供されるのは専門家（ドメインエキスパート）の理解に基づいたプロダクトではなく開発者の理解に基づいたもの
- 開発者がドメインを完全に理解した状態にするにはどうすればいい？

## DDDの目標
- ドメインエキスパートと開発者が同じメンタルモデルを共有し、そこからコードを生成する

## 単語の定義
- ドメイン
    - 守備一貫した知識の領域
        - 現れる単語が共通の概念を指している領域
    - ドメインエキスパートが専門としている物事
- 境界づけられたコンテキスト
    - DDDの用語
    - 解決手段における何らかの専門的知識の集合とそれが用いられるべき範囲のこと
    - なぜ境界づける？
        - ソフトウェアの世界では別々のサブシステムとして切り出し柔軟な進化をさせたいため
- ユビキタス言語
    - ビジネスドメインの共有メンタルモデルを定義する言語

## 具体的なガイドライン

### ビジネスイベントによるドメインの理解
- データ構造ではなくビジネスイベントやワークフローに焦点を当てる
    - データを持っているだけでなくそれを変換するのがビジネスの活動であるため
- 何らかのきっかけ（ビジネスイベントと呼ぶ）（時間、観察結果、外部からのトリガー）でデータの変換が行われる　という抽象化でビジネスを見る
    - このきっかけをドメインイベントとも呼ぶ

#### ドメインイベントの探索
- イベントストーミング
    - ステークホルダー＋ドメインエキスパートを可能な限り全員集めたワークショップを開催
- ビジネスイベントを書き出し、それが発火させるビジネスワークフローを書いていく
    - タイムラインに並び替えるなど
- イベントストーミングの効果
    - ビジネスの共通モデルを参加者全員が共有している
    - 全チームの把握
        - 不足しているビジネスワークフローの洗い出し
    - 要件のギャップが発見できる
    - チーム間の連携
        - チーム間のビジネスワークフローとビジネスイベントの連携がわかる
    - レポーティング要件の把握
        - ビジネスワークフロー内でレポート（証跡）を残す要件があるものの把握
#### 例:受注システムのドメイン探索
- 背景
    - 製造会社
    - 紙ベースの受注システムを改善したい

#### イベントを端まで広げる（ライフサイクル全体の把握）
- ビジネスワークフローの前後のつながりを辿っていき、要件の不足を見つける

#### コマンドの文書化
- コマンド
    - ドメインイベント（ビジネスイベント）を引き起こす依頼
        - 下の定義だと直接的な関係のみ見るとイベントに引き起こされているが、コマンドが引き起こしたビジネスワークフローがビジネスベントを引き起こしているのでこちらの性質に注目していそう
        - 人からの依頼が起点になっているもののみ？
- 性質
    - 常に「〜する」と書かれる
    - イベントを「〜された」
- 例
    - 顧客が注文書の受け取りを依頼する
    - 上司が作業を依頼する
- 流れ
    - イベント -> コマンドがトリガーされる（〜する） -> ビジネスワークフローが実行 -> ビジネスイベント（〜した）
- 全てのイベントがコマンドと紐づいていなくて良い
    - イベントが必ずしもコマンド（イベントを見た人間が行う依頼）と紐づくわけではないため

### ドメインをサブドメインに分割する
- 一つのドメインを複数のサブドメインに分割し、扱いを容易にする
    - 大きな単一のドメインをそのまま扱うのは困難であるため、分割統治する
    - 境界については後の境界づけられたコンテキストの設定の時だけ考える
- サブドメインの関係
    - サブドメイン間には親子関係、重複関係がある
- サブドメイン間は互いに重なり合う
    - 例:受注、配送、請求の業務ではそれぞれ「注文」などの概念やその取り扱いについて知っておく必要がある -> 受注サブドメイン、配送ドメイン、請求ドメインの重なり合いとして表現

### 各サブドメインのモデルを解決空間に作成する
- 問題空間（実世界）から解決空間（ソリューション内）を作成する
    - ドメインの必要な側面のみを抽出しそれらを解決空間に再作成する
- 問題空間内のドメイン、サブドメイン -> 解決空間内の「境界づけられたコンテキスト」と対応づける
    - 「境界づけられたコンテキスト」というのは解決手段における専門的知識を表しているため
    - 問題空間のドメインと解決空間の境界づけられたコンテキストは1:1でなくても良い
- 全ての境界づけられたコンテキストを重視する必要はない
    - 最もビジネス上の価値があるコンテキストに注力する
    - それ以外は外注などで切り出しても良い
        - この切り出しを容易にするためのサブドメインの分割でもある

#### コンテキストの境界設定のためのガイドライン
- ドメインエキスパートの会話で同じ言語の共有、同じ問題に焦点が当たっている場合はおそらく一つの境界づけられたコンテキストである
- 部門の境界や実際の境界に注目
    - 部門の境界 -> 管理者側が考えている境界
    - 実際の境界 -> 部門を超えて密接に業務をしている場合、同じ境界づけられたコンテキストを共有している可能性がある
- 境界を明確に設定することを強く意識する
    - 曖昧な境界はコンテキスト内の単語が指す意味を曖昧にし、実装を困難にするため
- 自律性（それぞれの境界づけられたコンテキストが別々に（他のコンテキストとの調整が必要なく）進化できる性質のこと）を意識する
- 摩擦のないビジネスワークフローの設計を意識
    - ビジネスワークフローを実行する中、複数の境界づけられたコンテキストと関わりブロックや遅延をもたらす場合はコンテキストの方の再設計を行うべき
    - ビジネスワークフローの摩擦のなさ > コンテキスト設計の煩雑さ

#### コンテキスト間の相互関係の図子
- コンテキストマップを作成し、コンテキスト同士がそれぞれの設計の詳細に巻き込まれることなく相互作用があることを伝える
- 有向グラフで表現する
- それぞれの関係間でのデータのやり取りのフォーマットの共通化は意識して行う

### ユビキタス言語の創造
- 解決手段とドメインエキスパートは同じモデルを共有しなければならない
- 概念（意味、それが行う振る舞いなど）と語彙のセットが少なくとも境界づけられたコンテキスの中では一致しているべき
    - ドメインエキスパートが何かを「注文」と呼んでいるなら解決手段内で「Order」という単語が現れ、モデルと同じように振る舞うべき
    - ドメインエキスパートのモデルの中で何かを表現していないものを設計の中に含めるべきでない
        - Factory, Helperなど
- この語彙をユビキタス言語という
    - プロジェクト固有になりがちなので、Wiki等で管理すべき
